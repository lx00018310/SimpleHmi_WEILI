# 架构阶段规范

架构阶段负责把规划输入转成最小系统结构。目标是降低不确定性，不是追求大而全。

## 必读

- `../SIMPLE_HMI.md`
- `../ENGINES.md`
- `10_deployment_control规范.md`
- `11_工程避坑清单.md`

## 架构输出

必须输出：

- 技术栈判定
- 模块边界
- 前后端通信方式
- 数据流
- 配置源
- 调试隔离策略
- deployment_control 评估结论
- Demo 阶段范围

## 技术栈判定

默认优先 TypeScript 后端。满足以下情况才优先 Java Spring Boot：

- 甲方明确要求。
- 企业系统对接重。
- 维护团队偏 Java。
- 旧系统复用明显降低风险。

AI 必须写出判定理由。

## 模块边界

建议最小边界：

- `web-hmi`：前端 HMI。
- `api`：后端 API。
- `contracts`：接口契约。
- `adapters`：硬件或外部系统适配接口。
- `deployment_control`：部署控制评估和接入。

## deployment_control 判断

架构阶段只做评估，不直接强套：

- 新项目、多节点、需要回滚：倾向接入。
- 复杂存量项目、多硬件混杂、排查链过长：谨慎接入。

## 禁止项

- 禁止同时支持两套主后端技术栈。
- 禁止为了“未来可能”添加大量抽象。
- 禁止把调试模拟放进正式模块。
- 禁止把部署控制器和业务逻辑耦合。

## 门禁

进入 Demo 阶段前必须满足：

- 技术栈已判定。
- 主数据流已画清或文字描述清楚。
- Mock 策略明确为默认禁用。
- deployment_control 已完成评估。
- Demo 只覆盖最小主链路。
