# 工程避坑清单

本清单沉淀工业 HMI / 工控前后端项目的通用风险，不绑定任何具体业务项目。

## 新项目经验

- 新项目可以逐步建立 deployment_control、发布包、健康检查、回滚、自愈。
- 配置、令牌、节点身份必须统一，不能多源并存。
- 控制器状态需要持久化，不能只存在内存。
- 发布包必须校验生产构建内容，尤其要检查未关闭的调试入口和 Mock UI。
- Windows 原生部署和 Linux Docker 部署要分开设计，不要跨平台硬套。

## 存量项目风险

- 复杂存量项目强套 deployment_control 可能增加排查链。
- 多硬件、多平台、多服务混杂时，生产链路越薄越好。
- 调试接口必须从正式控制接口拆出。
- 前端调试入口必须能关闭或删除。
- 配置要收口，不能多个文件或环境变量同时控制同一现场参数。
- 涉及真实设备动作时，必须经过真实联调验证。

## 共同原则

- 不要假成功。
- 不要静默降级。
- 不要把调试能力留在正式入口。
- 不要让多个配置源控制同一个现场事实。
- 不要用文档完成代替构建、运行、现场验证。
- 不要让部署控制器和业务逻辑互相污染。

## AI 使用方式

架构、调试、部署、验收阶段必须对照本清单，回答：

- 当前项目是新项目还是存量改造？
- deployment_control 是否会降低风险，还是增加排查链？
- 是否存在 Mock、假成功、降级、调试残留？
- 是否存在多配置源、部署不清、回滚不明的问题？
