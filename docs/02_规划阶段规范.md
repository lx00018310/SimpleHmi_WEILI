# 规划阶段规范

规划阶段只做一件事：把项目边界问清楚。不要写代码，不要设计复杂架构。

## 输入

需要用户或资料提供：

- 业务目标
- 现场工况
- 终端类型
- 屏幕分辨率和方向
- 硬件设备
- 外部系统
- 操作人员角色
- 成功标准
- 失败场景
- 甲方或运维约束

## 必问问题

AI 至少要确认：

- 这个系统控制什么、显示什么、记录什么？
- 哪些信号来自真实硬件？
- 哪些接口由第三方提供？
- 哪些动作有安全风险？
- 是否允许写 PLC 或控制设备？
- 是否要求 Java Spring Boot？
- 是否需要多屏、多节点统一发布？
- 是否需要 deployment_control？
- 是否允许做临时 Mock 调试能力？

## 输出

规划阶段输出：

- 需求边界清单
- 现场设备清单
- 外部接口清单
- 角色与权限清单
- 失败场景清单
- 待确认问题清单
- 下一阶段架构输入

## 禁止项

- 禁止默认假设硬件已可用。
- 禁止默认第三方接口稳定。
- 禁止默认可以 Mock。
- 禁止默认可以写 PLC 或控制真实设备。
- 禁止在边界不清时进入代码实现。

## 门禁

进入架构阶段前必须确认：

- 项目目标能用一句话描述。
- 至少列出真实硬件和外部系统。
- 至少列出 3 个失败场景。
- 明确是否有 Java / 甲方技术要求。
- 明确 deployment_control 是“需要评估”，而不是已默认启用或禁用。
